# Governance Gap Memo｜治理與信任設計說明

**專案名稱**　代辦人（Daibanren）— 勞資爭議可信 AI Agent
**團隊／案件方向**　04 政府服務：代辦到哪一步，由授權說了算
**一句話**　勞工遇到爭議時，Agent 代表本人蒐集證據、驗證完整性、產生申請書並協助送件，全程留下可舉證的軌跡。

---

## 1. 場景痛點

勞工在遭遇加班未給付、非法資遣等爭議時，最現實的困難不是「不知道自己的權利」，而是**蒐證與行政程序的門檻太高**：對話紀錄要自己整理、出勤資料要自己比對、申請書格式要自己研究，往往在情緒與資訊落差中錯過黃金處理期。讓 AI Agent 代為蒐證、草擬與協助送件可以大幅降低門檻，但也立刻帶來一個更根本的問題：**一個能讀取私人對話、能草擬法律文件、甚至能觸發送件的 Agent，要如何被信任？**

## 2. 六大信任要素如何落地

| 要素 | 本案設計 |
|---|---|
| **Principal** | Agent 僅代表單一委任人（勞工本人）行動，所有動作皆綁定案件編號與委任人身分，不代表雇主或第三方。 |
| **Authorization** | 建立案件時即產生明確授權書：允許「唯讀」對話與出勤資料、允許草擬文件；**禁止**修改原始證據、**禁止**自行送出申請書。允許與禁止事項在 UI 上以顏色標籤明確區分，而非隱藏在條款中。 |
| **Tool / Action** | Agent 可呼叫的工具被拆分為最小單位（讀取對話 / 讀取出勤 / 生成草稿），每個工具各自留痕，沒有一個「萬用」工具可以做完所有事，降低越權風險。 |
| **Policy Gate** | 「送出申請書」被定義為高風險動作，UI 上強制要求委任人勾選確認並手動點擊「本人核准並送出」，Agent 本身無法跳過此關卡，體現 human-in-the-loop 的強制性而非選配性。 |
| **Audit Log** | 每個動作在發生當下即計算 SHA-256 雜湊，並將前一筆紀錄的雜湊一併納入計算，形成鏈狀稽核紀錄；Demo 提供「模擬竄改證據」按鈕，讓評審實際看到雜湊比對失敗、竄改被即時偵測的過程，而非僅是文字宣稱。 |
| **Expiry / Revocation** | 授權書載明 7 天效期，逾期自動失效；委任人可在任何步驟按下「立即撤銷」，撤銷後 Agent 立即停止讀取、生成與送件等一切動作。 |

## 3. 這個 Demo 誠實揭露的邊界

- 雜湊鏈為**本地即時計算之真實 SHA-256**，並非部署於公開區塊鏈網路；VC（可驗證憑證）與 DID 為概念示意，尚未串接真實憑證發行機構。
- 申請書草稿為**預先準備之範本模擬 Agent 生成流程**，未呼叫外部 AI API 生成內容；正式產品應以真實 LLM 生成並強化幻覺與事實查核機制。
- 送出至勞工局調解窗口為**模擬流程**，未串接真實政府系統或身分驗證機制（如自然人憑證、TW FidO）。
- 所有對話紀錄、出勤資料、當事人與雇主名稱皆為**虛構合成資料**，不涉及真實個人或企業。

## 4. 若要走向正式產品，下一步會補上什麼

1. 以 DID／VC 發行可驗證的「委任授權憑證」，取代目前的應用內設定畫面，讓授權本身可被第三方（如勞工局、律師）獨立驗證。
2. 稽核雜湊鏈上鏈至具時間戳公證效力的服務（如 Timestamping Authority 或公開帳本），取得對外可信的第三方佐證。
3. 導入人工審查／律師複核流程於 Policy Gate 之前，降低 AI 生成內容之法律風險。
